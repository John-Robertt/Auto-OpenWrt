# 文档先于代码规则

## 基本原则

Auto-OpenWrt 采用文档先于代码的开发规则。产品目标、架构边界、行为规格和验收标准必须先被记录和确认，代码只负责实现已确认的设计。

这条规则用于避免在实现阶段临时做产品或架构决策，保护后续修改自由，并让每个实现都能被追溯、评审和验证。

## 开发准入

开发任何功能前，必须满足以下条件：

- 有对应的 accepted 产品目标或用户需求。
- 有明确的 accepted 架构归属和模块边界。
- 有可实现的 accepted 规格说明。
- 有可执行的验收标准。
- 如存在关键取舍，已有 accepted ADR。
- 对应实施计划为 accepted。

不满足以上条件时，不进入代码实现。

## 变更规则

- 涉及模块边界、数据结构、配置格式、CLI 行为、状态流转的改动，必须先更新 `02-architecture/` 或 `03-specs/`。
- 出现两个以上可行方案且存在取舍时，必须先写入 `04-decisions/`，再进入实现。
- 实现计划只能引用已确认文档，不能在实现计划中新增未经确认的产品或架构决策。
- 代码实现过程中发现文档与现实冲突时，先暂停实现，更新文档并确认新决策，再继续开发。
- 每个功能完成后，验证结果要回写到对应 implementation plan 或验收记录中。

## 职责边界

- 产品文档只写目标、需求、产品职责和验收标准。
- 架构文档只写系统结构、模块边界、依赖方向、核心模型和流程。
- 规格文档写可实现行为、输入输出、状态、错误处理和文件契约。
- ADR 记录关键取舍和决策结果。
- 实施计划拆解开发步骤，不新增产品或架构决策。

## 实现冻结条件

出现以下任一情况时，必须暂停代码实现，只能继续做文档澄清或规格加固：

- 目标功能依赖的规格仍为 `draft` 或 `proposed`。
- 实施计划仍为 `draft` 或 `proposed`。
- 文档依赖图存在环。
- 实现者需要自行决定数据格式、状态枚举、命令参数、错误语义或持久化路径。
